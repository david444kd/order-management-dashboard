``# Order Management Dashboard

Прототип TMS-модуля для управления грузовыми заказами. Бэкенда нет — всё работает через mock API в localStorage, чтобы можно было оценить UX и зафиксировать data-контракт до старта бэкенда.

**Демо:** https://order-management444.netlify.app/

## Запуск

```bash
npm install
npm run dev
```

```bash
npm run test    # unit-тесты
npm run lint
npm run build
```

## Роуты

| Роут | Что делает |
|---|---|
| `/orders` | Таблица с фильтрами, сортировкой, пагинацией и row actions |
| `/orders/new` | Draft workspace: до 5 вкладок, автосохранение, восстановление после перезагрузки |
| `/orders/:id` | Детали заказа, стопы, история статусов |
| `/orders/:id/edit` | Редактирование — только для `pending` |

## Архитектура

### Feature-Sliced Design

```
src/
├── app/          # роутер (lazy routes), провайдеры``
├── pages/        # тонкая композиция экранов
├── widgets/      # крупные блоки: OrdersTable, DraftWorkspace
├── features/     # сценарии: filter-orders, change-order-status, manage-drafts…
├── entities/     # доменная модель + API: order, carrier, draft
└── shared/       # mock client, UI-примитивы, хуки, Zod-схемы
```

Каждый слой импортирует только из слоёв ниже. При росте кодовой базы это важно — на ~40 фичах flat-структура начинает разваливаться.

### Zustand + TanStack Query

Два разных вида состояния — два разных инструмента.

**TanStack Query** — серверное состояние (заказы, перевозчики). Кеш, инвалидация, retry, фоновый рефетч — всё из коробки. `staleTime: 30s`.

**Zustand** — клиентское состояние: фильтры, сортировка, активная вкладка черновика. Это состояние никогда не «устаревает», query-кеш для него семантически неправильный выбор.

Redux Toolkit не рассматривал — при таком объёме это лишний boilerplate. Если появятся сложные cross-feature флоу или offline-конфликты, RTK станет логичным следующим шагом.

### Mock API

Симулирует реальный бэкенд: задержка 300–800ms, ~5% случайных ошибок, полный CRUD по заказам. Данные сидируются в localStorage при первом запуске.

## Что бы я улучшил

**Состояние таблицы в URL** — фильтры и сортировка сейчас в Zustand, сбрасываются при перезагрузке. В query params это шаринг ссылки, кнопка «назад» и deep link из внешних систем. Для диспетчеров в TMS это реальный use case, не просто красота.

**Оптимистичные обновления** — после смены статуса таблица уходит в loading. Для самого частого действия в системе это заметно. TanStack Query позволяет обновить кеш немедленно и откатиться при ошибке — минимальный риск, ощутимый выигрыш.

**MSW вместо mock в localStorage** — сейчас весь mock-код попадает в production bundle, потому что живёт внутри приложения. MSW работает на уровне Service Worker, полностью вне приложения. Fetch-вызовы остаются настоящими, переход на реальный бэкенд — просто удалить хендлеры. Плюс MSW работает в Vitest, что открывает интеграционные тесты без моков на уровне модулей.

**E2E на ключевые флоу** — юнит-тесты не ловят интеграционные баги. Playwright-сценарий «создать draft → submit → заказ появился в таблице» поймал бы рассинхрон между инвалидацией кеша и localStorage — это то, что юнит-тестами не проверить.

**Devtools** — React Query Devtools и Zustand devtools не подключены. Оба tree-shaking friendly, в production не попадают. Без них отлаживать «почему кеш не инвалидировался» — только через console.log.
